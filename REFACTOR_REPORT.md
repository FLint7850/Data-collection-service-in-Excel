# Отчёт о полном рефакторинге

## Что было

- `app.py` — 7 733 строки: Flask, конфигурация, БД, парсинг, нормализация, экспорт, фоновые задачи, расписание и API в одном файле.
- `static/js/app.js` — 2 741 строка: состояние, DOM, API, настройки, новинки, рабочая область и события в одном файле.
- Инициализация хранилища могла повторно проверять схему БД и перечитывать настройки почти на каждом запросе.
- Пути к шаблонам, статике и SQLite зависели от текущей структуры запуска.
- Production-запуск не был отделён от режима разработки.

## Новая структура

```text
app.py                         # совместимая точка входа для разработки
wsgi.py                        # WSGI entrypoint
run_production.py              # production-запуск через Waitress
parser_app/
  runtime.py                   # Flask, конфигурация, пути, общее runtime-состояние
  bootstrap.py                 # совместимость между вынесенными модулями
  server.py                    # запуск приложения
  routes/                      # маршруты по предметным областям
  services/                    # бизнес-логика и инфраструктурные сервисы
static/js/app/
  core.js                      # DOM, состояние и общие функции
  settings.js                  # настройки
  news.js                      # мониторинг новинок
  workspace.js                 # рабочая область
  events.js                    # обработчики интерфейса
```

## Выполненные изменения

- Backend разделён на маршруты и сервисы без удаления существующих endpoint-ов.
- Frontend разделён на пять упорядоченных модулей без изменения текущего DOM-контракта.
- `ensure_storage()` сделан идемпотентным и потокобезопасным: дорогая инициализация выполняется один раз на процесс.
- Добавлено кеширование регулярного выражения брендов и сокращены повторные вычисления при нормализации моделей.
- Пути к шаблонам, статике и БД сделаны явными; добавлена переменная `DATABASE_PATH`.
- Добавлены отдельные production-настройки, проверка секретного ключа и запуск Waitress.
- Основные и тяжёлые необязательные парсеры разделены по файлам зависимостей.
- Добавлены архитектурная документация, инструкция развёртывания и автоматические тесты.
- Удалена мёртвая/лишняя логика и production-артефакты не включаются в чистую сборку.

## Важное ограничение production

Планировщик и состояние активных задач пока хранятся в памяти процесса. Поэтому приложение следует запускать **одним процессом с несколькими потоками**. Несколько независимых WSGI workers приведут к отдельным планировщикам и несогласованному runtime-состоянию. `run_production.py` уже использует подходящую схему Waitress.

## Команда запуска

```bash
python run_production.py
```

Перед запуском скопируйте `.env.example` в `.env`, задайте уникальный `FLASK_SECRET_KEY`, учётные данные и production-параметры. Полная инструкция находится в `docs/DEPLOYMENT.md`.

## Граница гарантии

Статические, модульные, сравнительные и smoke-проверки проходят. Полностью гарантировать отсутствие ошибок на всех реальных сайтах-донорax невозможно без staging-прогона с production-сетью, браузерными движками, прокси и фактическими данными каждого источника. Перед переключением боевого трафика рекомендуется один полный staging-цикл сбора, сравнения, экспорта, расписания и почтовых уведомлений.
